# Tesla Foundation Architecture & Rules

This document outlines structural catches and strict architectural rules mechanically enforced across the codebase. Any AI agent or developer stepping onto the codebase MUST mathematically adhere securely to these patterns natively!

## 1. Monorepo (NPM Workspaces)
This project is an **NPM Workspace Monorepo**. Do NOT run separated isolated `npm install` executions across subfolders blindly.
- All core Typescript interfaces specifically map into the `@tesla/shared` workspace located at `/shared/`.
- Both `frontend` (React) and `backend` (Express) strictly pull shared types globally preventing disjointed UI components directly!
- If modifying core definitions natively (like `user.ts`), strictly rebuild `@tesla/shared` first mappings statically.

## 2. strict TypeScript Native
The entire ecosystem scales recursively on Strict TypeScript.
- **Zero 'any' Policy**: The use of `any` is strictly prohibited. Use `unknown`, `never`, or explicit interfaces/types instead.
- **Shared Type Hub**: If a type is shared between `frontend` and `backend`, it MUST reside in the `@tesla/shared` workspace (`/shared/`). Do not duplicate interfaces.
- **Backend Constraints**: Transpiles natively compiling directly to CommonJS via `backend/tsconfig.json` mappings explicitly towards a `/dist` bucket. Do not inject raw dynamic `require()` mappings dynamically out-of-scope; utilize ES6 imports seamlessly mapped.
- **Schema Mapping**: All Mongoose `.model` structures natively inherit and extend directly from `@tesla/shared` definitions avoiding duplicating `IUser` mapping arbitrarily natively.

## 3. Render.com Infrastructure (render.yaml)
Deployments are entirely processed algorithmically via `render.yaml` Infrastructure as Code.
- Build commands MUST run from the monolithic Application Root `/`, executing identically: `npm install --include=dev && npm run build -w @tesla/shared && npm run build -w [service]` strictly building the Monorepo dependencies globally.
- The `--include=dev` flag is EXTREMELY IMPORTANT on Render.com. Because Render defaults to `NODE_ENV=production`, standard `npm install` skips devDependencies (like `@types/jsonwebtoken`). Without this flag, the TypeScript build (`tsc`) will fail.
- Avoid forcing static `cd frontend && npm install` scripts manually bypassing workspaces internally inside Render commands!

## 4. Environment Variables Mapping (Graceful Scaling)
The `MONGODB_URI` parameter safely establishes real Native DB interactions seamlessly targeting AWS Clusters (or Atlas clusters) dynamically.
- `backend/src/loaders/mongoose.ts` safely traps the execution natively gracefully logging `⚠️ No MONGODB_URI found.` bypassing a crash loop!
- Never natively track or upload `.env` containing connection passwords to Github manually! They must be securely injected via Render's **Environment Variable Dashboard** manually!

## 5. CSS Modules Pattern
- All frontend UI elements strictly bypass global stylesheet pollution mapping rigorously to isolated `FileName.module.css`. TailWind CSS natively is mathematically blocked across aesthetic preferences explicitly requested earlier natively!

## 6. Stateless Token Authorization (JWT)
- Sessions are strictly maintained via `jsonwebtoken` algorithmically generating encrypted `token` strings across successful login states structurally logging rigid 1-hour TTLs dynamically.
- The React Frontend mechanically taps `localStorage.getItem('token')` executing `fetch` against `/api/auth/me` to hydrate persistent states inherently stopping refresh-bugs.
- If rewriting custom hooks or API fetch scripts, implicitly guarantee `<Authorization: Bearer>` mappings organically into Http Headers gracefully matching the backend.

## 7. Mandatory Internationalization (i18n)
- **Zero Hardcoding**: All user-facing strings MUST be localized via the `useLanguage` hook and `t()` function. 
- **Registry Compliance**: Any new string mapping MUST be added to `frontend/src/i18n/locales.ts` for all supported languages (`en`, `es`, `fr`).

## 8. Continuous Testing & Regression Prevention
- **TestCase Evolution**: Every logic modification, core structural change, or UI workflow update MUST be accompanied by a corresponding update or addition of a test case.
- **Verification Stack**: Use **Jest** for unit/hook logic (`__tests__/*.test.tsx`) and **Playwright** for E2E user-flow verification (`e2e/tests/*.spec.ts`).
